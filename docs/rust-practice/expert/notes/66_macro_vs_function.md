# 66 Macro vs Function - 笔记

## 本题会用到的前置知识点

- 函数签名设计。
- 字符串格式化。
- 宏的基本语法。

## 核心结论

- 能用函数清楚表达时，优先函数。
- 宏适合语法级复用，不是默认抽象工具。
- 选择标准：可读性、可调试性、维护成本。
- 如果只是接收普通值、返回普通值，函数通常更合适。
- 宏的合理理由通常是：需要可变数量参数、特殊语法、避免重复写结构性代码，或必须在类型检查前生成代码。

## 需要补充看懂的知识

### 函数的边界更清楚

```rust
fn to_kv_line(k: &str, v: &str) -> String {
	format!("{}={}", k, v)
}
```

这个签名已经说明了三件事：

- 输入是两个可借用的字符串片段。
- 输出是拥有所有权的 `String`。
- 调用时走普通类型检查。

这类逻辑不需要宏。函数更容易测试、重构、搜索和调试。

### 宏的成本

宏不是不能用，而是有成本：

- 错误信息可能指向展开后的代码，初学者不容易读。
- IDE 跳转、重构、调试通常不如函数直接。
- 宏匹配的是语法片段，不是普通参数类型，接口更难约束。
- 宏体复杂后，读者要同时理解“匹配规则”和“展开后代码”。

所以本题的结论不是“宏不好”，而是：没有语法级收益时，函数是更小的工具。

### 什么情况宏才更有理由

更合理的宏场景：

```rust
println!("{} {}", a, b); // 可变参数数量和格式串语法
vec![1, 2, 3];           // 可变参数数量，生成 Vec 初始化代码
assert_eq!(left, right); // 捕获表达式文本，给出更好的失败信息
```

这些都不是普通函数能同样自然表达的场景。相比之下：

```rust
to_kv_line("mode", "dev")
```

只是普通值转换，用函数就够。

### 判断顺序

写宏前先问：

1. 能否用普通函数表达？
2. 如果能，函数版本是否已经清楚？
3. 宏是否减少了真实重复，还是只是让代码看起来更高级？
4. 宏带来的调试和维护成本是否值得？

本题最终保留函数版本，就是练这个判断。

## 核心示例

```rust
fn to_kv_line(k: &str, v: &str) -> String {
    format!("{}={}", k, v)
}

fn main() {
    println!("{}", to_kv_line("mode", "dev"));
}
```

理解重点：这个场景函数足够清晰，宏收益很小。

## 常见坑

- 为了"看起来高级"而滥用宏。
- 不比较函数方案就直接上宏。
- 把“少写几个字符”当成使用宏的理由。
- 忽略函数签名本身提供的类型约束和可读性。

## 回看问题

- 这个问题必须要语法级能力吗？
- 换成函数后，是否更容易测试和维护？
- 宏版本是否真的比函数版本表达了更多信息？
