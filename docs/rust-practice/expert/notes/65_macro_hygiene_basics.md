# 65 Macro Hygiene Basics - 笔记

## 本题会用到的前置知识点

- 局部变量作用域。
- 表达式求值。
- 宏基础匹配与展开。

## 核心结论

- 宏内部临时变量应尽量避免干扰调用方命名。
- 宏参数表达式要注意“只求值一次”。
- 读代码时要先看展开后是否清晰。
- Rust 的 `macro_rules!` 有卫生性保护，但仍要避免写出依赖调用方局部名字的宏。
- 用块表达式包住宏体，可以把内部临时变量限制在局部作用域里。

## 需要补充看懂的知识

### 卫生性解决的是“名字不要意外串线”

宏会展开成调用处的代码。一个自然担心是：宏内部定义的变量会不会和调用方已有变量冲突？

```rust
macro_rules! measure_len {
	($input:expr) => {{
		let __value = $input;
		__value.to_string().len()
	}};
}
```

调用方也可以有自己的变量：

```rust
let __value = "caller";
let n = measure_len!("rust");
```

入门阶段可以这样理解：宏内部的 `let __value` 应该被当成宏展开内部的临时变量，不应改变调用方同名变量的含义。实际的宏卫生规则更细，但本题先掌握这个直觉即可。

### 为什么要把 `$input` 先存起来

宏参数是语法片段。如果在展开体里多次写 `$input`，输入表达式就可能被执行多次：

```rust
// 错误思路：$input 出现两次
// macro_rules! bad {
// 	($input:expr) => {{
// 		$input.to_string().len() + $input.to_string().len()
// 	}};
// }
```

如果调用方传入的是有副作用或有成本的表达式，问题就很明显：

```rust
// bad!(make_string())
```

`make_string()` 可能会被调用两次。更稳的写法是先求值一次：

```rust
let __value = $input;
```

后面只使用 `__value`。

### 宏体不要隐式依赖调用方变量

下面这种宏很脆弱：

```rust
// 错误思路：宏体依赖调用方刚好有变量 prefix
// macro_rules! add_prefix {
// 	($input:expr) => {
// 		format!("{}{}", prefix, $input)
// 	};
// }
```

调用方如果没有 `prefix`，展开后就编译失败；如果有，也很难从调用点看出宏依赖了它。更清楚的设计是把依赖显式作为参数传入：

```rust
// add_prefix!(prefix, value)
```

### 临时变量名仍要认真取

卫生性不是让你随便写复杂宏。内部临时变量最好：

- 放在块表达式里。
- 名字能表达用途。
- 不出现在宏的对外语义里。
- 不依赖调用方同名变量。

## 核心示例

```rust
macro_rules! measure_len {
    ($input:expr) => {{
        let __value = $input;
        __value.to_string().len()
    }};
}

fn main() {
    let tmp = 10;
    let n = measure_len!("rust");
    println!("{tmp}, {n}");
}
```

理解重点：宏内部变量名与调用方同名变量不应造成语义混乱。

## 常见坑

- 在宏里多次使用 `$input`，导致重复求值。
- 宏体隐式依赖调用方环境。
- 不用块表达式包住宏体，导致展开后的局部变量范围不清楚。
- 以为卫生性可以替代清晰的宏接口设计。

## 回看问题

- 输入表达式是否只执行了一次？
- 展开后代码在作用域上是否清楚？
- 宏依赖的所有信息是否都从参数传入？
