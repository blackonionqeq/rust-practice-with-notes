# 51 Lock Scope and Deadlock - 补充笔记

## 背景知识

### 什么是死锁

死锁（deadlock）指两个或多个线程互相等待对方手里的锁，谁也动不了，程序卡死。最常见的触发原因是**加锁顺序不一致**。

### 经典的两把锁场景

假设有两个共享计数器 `A` 和 `B`，各用一把 `Mutex` 保护：

```rust
use std::sync::{Arc, Mutex};

let a = Arc::new(Mutex::new(0));
let b = Arc::new(Mutex::new(0));
```

- 线程 1：先锁 `A`，再锁 `B`。
- 线程 2：先锁 `B`，再锁 `A`。

如果线程 1 拿到 `A` 的同时，线程 2 拿到了 `B`，然后两边都去等对方手里的那把锁——死锁。

### 一致的加锁顺序就能避免

只要所有线程**总是按相同顺序**获取同一组锁，就不会出现循环等待：

```rust
// 安全：所有需要同时动 A 和 B 的地方，都先锁 A 再锁 B
fn update_both(a: &Mutex<u32>, b: &Mutex<u32>) {
    let mut a_guard = a.lock().unwrap();
    let mut b_guard = b.lock().unwrap();  // 固定顺序：永远 A -> B
    *a_guard += 1;
    *b_guard += 1;
}
```

给锁排一个全局顺序（比如按名字、按地址、按固定获取顺序），所有人遵守，就能消除这类死锁。

### 锁的作用域要短

guard 在 drop 时释放锁。如果持锁期间还去做别的事，持锁时间拉长，死锁和争用的概率都会上升。把锁的获取和释放限制在最小的 `{}` 块里：

```rust
// 好：用完立刻放锁
{
    let mut a_guard = a.lock().unwrap();
    let mut b_guard = b.lock().unwrap();
    *a_guard += 1;
    *b_guard += 1;
}   // 两把锁都在这里释放
join_or_slow_work();   // 不再持有任何锁
```

### 加锁顺序不是万能的

一致的加锁顺序能解决"多把锁循环等待"的死锁，但防不住所有并发问题：

- **单把锁递归请求**：同一把 `Mutex` 不可重入，同一个线程里再次 `lock` 会死锁。要拆分函数、缩小锁范围。
- **持锁期间等待外部事件**：拿着锁去等另一个线程或 IO，容易死锁。
- **活锁 / 饥饿**：没死锁但一直推进不了，需要更高级的并发设计。

真正彻底的解法往往是**减少锁的数量和持有时间**，或者改用消息传递，从根上避开"同时持有两把锁"。

## 本题覆盖

Practice recognizing and avoiding lock scope problems.

## 需要重点理解

- 死锁的典型成因是多个线程以**不一致的顺序**获取同一组锁。
- 给锁定一个全局顺序、所有线程遵守，就能消除循环等待。
- 锁 guard 在 drop 时释放，作用域越短越安全。
- 缩小锁范围、用消息传递替代多锁，是从根本上降低死锁风险的方法。

## 常见坑

- 两个函数以相反顺序获取同一组锁，多线程下偶发死锁（且很难复现）。
- 持锁期间调用别的函数，那个函数里又去拿锁，不知不觉就嵌套加锁。
- 以为"我加了锁就安全了"，却忽略了加锁顺序。
- 对同一把 `Mutex` 在同一线程内请求两次，直接死锁。

## 回看问题

- 你更新的两个计数器，所有代码路径里都是同一个加锁顺序吗？
- 每把锁的 guard 是不是在用完后第一时间就 drop 了？
- 如果把其中一把锁换成消息传递，这个设计会不会更简单？
