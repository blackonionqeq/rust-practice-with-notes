# 13 Lifetime Longest - 补充笔记

## 本题覆盖

- 生命周期参数。
- 输入引用和输出引用的关系。
- 返回字符串切片。
- 避免返回局部变量引用。

## 需要补充

### 生命周期不是延长引用寿命

```rust
fn longest<'a>(first: &'a str, second: &'a str) -> &'a str
```

这不是让 `first` 或 `second` 活得更久。它只是告诉编译器：

返回值的引用来自 `first` 或 `second`，并且不能比两者中较短的那个活得更久。

### 为什么需要标注

函数可能返回 `first`，也可能返回 `second`。编译器需要知道输出引用和输入引用之间的关系。

当只有一个输入引用时，很多情况下可以省略生命周期；当有多个输入引用且返回其中之一时，通常需要显式标注。

### `&'a str` 中的 `'a`

`'a` 是一个生命周期参数名，名字本身没有特殊意义。也可以叫 `'text`，但惯例常用短名字。

同一个 `'a` 出现在多个位置，表示这些引用之间存在同一组生命周期约束。

### 不能返回局部变量的引用

错误示例：

```rust
fn bad<'a>() -> &'a str {
    let text = String::from("rust");
    &text
}
```

`text` 在函数结束时被释放，返回它的引用会悬垂，所以 Rust 禁止。

### 返回切片避免分配

本题返回 `&str`，说明结果是输入的一部分借用，不创建新字符串。若返回 `String`，就需要分配或移动所有权，练习重点会变成另一个主题。

## 常见坑

- 以为生命周期标注能让局部变量逃出函数。
- 在 `longest` 里创建新 `String` 再返回引用。
- 不理解返回值只能在两个输入都还有效时使用。

## 回看问题

- 生命周期标注描述的是“关系”还是“时长”？
- 为什么两个输入引用时，返回引用常常需要显式生命周期？
- 为什么不能返回函数内部新建 `String` 的引用？
